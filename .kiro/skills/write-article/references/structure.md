# structure.md — 글 구조와 발행 규칙

`voice.md`가 "어떻게 말하는가"라면 이 문서는 "어떻게 배치하는가"다.
front-matter·콜아웃·이미지 규칙은 리포의 `docs/writing-articles.md`를 그대로 따른다.

---

## 글 유형별 구조 템플릿

기존 16편은 대략 다섯 유형으로 나뉜다. 인터뷰에서 정한 유형에 맞춰 골격을 잡는다.
소제목 개수·깊이는 주제에 맞게 조절하되, **서문의 동기**와 **결론의 태도**는 유형별 특징을 지킨다.

### 고찰형 (개념을 깊게 파고드는 글)
예: `reflections-on-react-re-rendering`, `react-redux-ecosystem`
```
# 제목
서문 — 이 질문/의문을 품게 된 계기 (동료의 말, 실무 중 마주친 궁금증)
#### 배경 개념 정리
#### 파고들기 1 (공식문서/소스코드 인용하며 검증)
#### 파고들기 2
#### 결론? — 단정하지 않는다. 남은 의문·솔직한 찝찝함을 공유
```
- 소스코드·공식문서 링크를 가장 촘촘히 건다.
- 결론 제목에 물음표를 달기도 한다("결론?"). 답을 못 냈으면 못 냈다고 쓴다.

### 튜토리얼형 (따라 하며 문제를 해결하는 글)
예: `next-deploy-with-firebase-1`, `handle-jwt-with-axios-instance`, `how-to-convert-markdown-to-json`
```
# 제목
서문 — 어떤 실무 상황에서 이걸 하게 됐는지 (배포 환경 이전, 멘티 질문 등)
#### 전제/버전 명시 (Dependency Version 등)
#### 단계별 소제목 (영문 소제목 OK: "Setup Firebase", "Cookie Setting")
    - 각 단계마다 실무 코드블록 + 한국어 주석 + 왜 이렇게 하는지 설명
#### 마무리 — "본인 프로젝트에 맞게 변경해 쓰자", 다음 글 예고
```
- 실제로 마주친 에러 메시지·버전·해결 과정을 그대로 남긴다.
- 함정/주의는 `[!note]`로 강조한다.

### 회고형 (경험·성장을 돌아보는 글)
예: `what-i-learned-out-of-junior`
```
# (센치한 느낌으로) 제목
서문 — 이 회고를 쓰는 시점과 이유
#### 배운 점 1 (구체적 에피소드 + PM/동료의 말 인용)
#### 배운 점 2
#### 배운 점 3
### 결론 — 스스로에게 던지는 다짐, 열린 마무리
```
- 감정과 솔직함이 전면에 나온다. 취소선 자조가 가장 많이 쓰이는 유형.

### 오피니언형 (주장을 펼치는 글)
예: `importance-of-re-rendering-optimization`
```
# 질문형/도발형 제목 ("여러분들은 리랜더링 최적화는 얼마나 중요하게 여기시나요?")
서문 — 통념을 먼저 인정
본문 — 데이터/벤치마크/통계로 반론 (수치·이미지·외부 통계 인용)
독자에게 질문 던지기
결론 — "내가 하고 싶은 말은 ~가 아니다"로 오해를 차단하고 진짜 주장을 놓는다
```

### 히스토리형 (기술의 역사·흐름을 정리하는 글)
예: `browser-war`
```
# 제목
서문 — 지금의 현상에서 출발해 과거로 거슬러 올라가는 훅
#### 시대순 소제목 (태동 → 1차 → 2차 → 현재)
    - 에피소드·뒷이야기·이미지 풍부하게
마무리 — 앞으로의 전망, 다음에 다루고 싶은 주제 예고
```

## front-matter 규칙

본문 최상단에 `---`로 감싼 YAML 블록. 5개 필드 전부 필수.

```markdown
---
title: "제목 (콜론·특수문자 있으면 큰따옴표로 감쌀 것)"
description: SEO meta description으로 쓰이는 요약. 기존 글은 영문 요약 관례가 많음 — 사용자에게 확인.
keywords: ["키워드1", "키워드2"]
category: react
created_at: 2024-11-07 01:06
---
```

| 필드 | 규칙 |
| --- | --- |
| `title` | 글 제목. 콜론(`:`)·특수문자 포함 시 반드시 큰따옴표 |
| `description` | 검색 노출용 요약. 기존 글은 영문 요약이 다수 — 국문/영문 여부는 사용자 확인 |
| `keywords` | 문자열 배열. 상세 페이지에서 category와 병합됨 |
| `category` | **파일이 위치한 폴더명과 일치** (`develop`/`next`/`react`/`remix`/`web` 또는 신규) |
| `created_at` | `YYYY-MM-DD HH:mm`. 정렬·예약발행 기준 |

- `index`/`path`/`readingTime`은 빌드가 자동 주입한다. **직접 쓰지 않는다.**

## 예약 발행

프로덕션 빌드에서는 `created_at`이 도래하지 않은 글은 생성되지 않는다(KST 기준, UTC로 −9h 변환 비교).
미래 시각을 넣으면 그 시각 이후 배포부터 노출된다. 로컬 `yarn dev`에서는 필터가 없어 미래 글도 미리 보인다.
기본은 현재 시각으로 두되, 예약 발행이 필요하면 사용자에게 목표 시각을 확인한다.

## 콜아웃 `[!note]`

`[!note]`로 시작하는 문단은 `💡` 콜아웃 박스로 변환된다. 함정·주의·보충 설명에 쓴다.
문단 내 개행이 필요하면 `[break]` 토큰을 쓴다.

```markdown
[!note] 여기에 강조할 내용을 쓴다.
```

## 이미지

- 저장 위치: `content/article/<category>/image/<slug>/파일명`
- 마크다운 참조: 마크다운 파일 기준 상대경로 `image/<slug>/파일명`
  ```markdown
  ![flux-architecture.png](image/react-redux-ecosystem/flux-architecture.png)
  ```
- 지원 확장자: `png`, `jpg`, `jpeg`
- 빌드 시 `public/`으로 복사된다. 새 이미지 추가 후엔 개발 서버 재시작 또는 `yarn generate` 필요.
- 이미지 출처가 있으면 `[이미지 출처](url)`를 이미지 근처에 남기는 게 이 블로그 관례다.

## 코드블록

- 언어 태그를 붙인다(`tsx`, `jsx`, `json`, `bash` 등).
- 실무 코드를 쓰되 생략은 `/* ... */`, 의사코드는 "pseudo code"라고 주석으로 밝힌다.
- 주석은 한국어로, 왜 그렇게 하는지를 설명한다.

## 특수 인터랙티브 컴포넌트 (드묾)

지도·차트 같은 클라이언트 전용 컴포넌트는 `app/routes/article_.$category.$title/SpecificContent/index.tsx`의
`componentMap`에 slug를 키로 등록해야 렌더링된다. 일반 글에는 불필요하다.

## 발행 전 체크리스트 (5단계에서 검증)

- [ ] `content/article/<category>/<slug>.md` 생성 (파일명 = slug, 영문 소문자+하이픈)
- [ ] front-matter 5개 필드(`title`/`description`/`keywords`/`category`/`created_at`) 작성
- [ ] `category` 값이 폴더명과 일치
- [ ] 이미지는 `image/<slug>/`에 두고 상대경로로 참조
- [ ] voice.md의 AI향 제거 체크리스트 통과
- [ ] `yarn dev`로 목록/상세/이미지 렌더링 확인하도록 사용자에게 안내
