# examples.md — ✓/✗ 실제 문장 대조

기존 아티클에서 뽑은 진짜 문장(✓)과, 같은 내용을 AI가 흔히 쓰는 방식(✗)을 나란히 둔다.
초안을 쓸 때 ✗ 쪽으로 흐르고 있지 않은지 대조한다. ✓는 모두 실제 게시글 인용이다.

---

## 서문 — 동기로 연다

✓ (firebase 배포 글)
> "회사에서 vercel로 배포중인 프로젝트를 firebase로 배포환경을 변경해야 할 일이 생겼다. 우리 회사는 원래 firebase와 gcp를 사용하고 있었기에 관리포인트 일원화가 목적이다. 처음에는 단순히 배포환경만 변경해주면 끝날 일 같았으나, 생각보다 vercel에서 자동으로 관리해주는 포인트가 많았기에 시행착오를 겪었다."

✗ (AI향)
> "이 글에서는 Next.js 프로젝트를 Firebase에 배포하는 방법을 알아보겠습니다. Firebase는 강력한 호스팅 기능을 제공하는 플랫폼입니다. 다음 단계를 따라 진행해 보겠습니다."

차이: ✓는 "나/우리 회사"의 구체적 상황과 시행착오가 있다. ✗는 누구나 쓸 수 있는 소개문이다.

## 결론 — 이 대비가 가장 중요하다

✓ 열린 결론 + 솔직함 (re-rendering 고찰 글)
> "블로그를 운영하면서 이번 아티클에 가장 많은 시간을 쏟았고, 지금처럼 찝찝할 수 없다. ... 결국 Render가 비싼가. Commit이 비싼가. ... 어떤 선택을 내리든 스스로가 납득할 수 있고, 같이 일하는 동료를 설득할 수 있는 근거가 동반되어야 한다고 생각한다. 이번 아티클에서 그 근거의 편린이라도 제공이 되었길."

✓ 오해 차단 후 진짜 주장 (오피니언 글)
> "내가 하고 싶은 말은 '리랜더링 최적화가 중요하지 않다.'가 아니다. ... 주객을 전도해선 안된다. 무엇을 위해 최적화가 필요하다고 생각하는지, ... 그리고 그것이 본인이 판단해도 되는 것인지 다시 한번 생각해보자."

✗ AI향 결론 (실제로 초기 글 react-redux-ecosystem에 나타난 패턴 — 반면교사)
> "결론적으로 React에서 전역 상태 관리는 확장 가능하고 유지 관리가 용이한 애플리케이션을 구축하는 데 필수적이다. ... 2024년의 최신 트렌드를 반영하여 경량 라이브러리와 개선된 패턴을 활용하면 애플리케이션 성능과 개발자 경험을 크게 향상시킬 수 있다."

세 번째가 왜 나쁜가: "필수적이다", "크게 향상시킬 수 있다"는 근거도 개성도 없는 일반론이다.
누가 써도 똑같고, "나"의 고민이 없다. **결론에서 이런 문장이 나오면 즉시 다시 쓴다.**

## 솔직한 한계 인정

✓ "각 과정에서 React가 어떻게 동작하는지를 배울 수 있었으나 처음 느꼈던 의문을 해결하지 못하였다."
✓ "정확히 어떤 feature가 지원되지 않는 것 까지는 분석하지 못했다."
✗ "이로써 모든 동작 원리를 완벽하게 이해할 수 있었습니다."

모르는 건 모른다고, 못 한 건 못 했다고 쓴다. 이 솔직함이 신뢰와 사람 냄새를 만든다.

## 실무 코드 + 한국어 주석

✓
```tsx
instance.interceptors.response.use(
  (response) => response,
  // 에러가 발생했을 시
  async (error) => {
    // 에러가 발생한 request를 임시 저장한다.
    const originalRequest = error.config;

    // 401 에러가 아닌 에러는 요청 실패임으로 reject한다.
    if (error.response?.status !== 401) {
      return Promise.reject(error);
    }
    /* ... */
  }
);
```

주석은 한국어로, "무엇을"보다 "왜"를 설명한다. 생략은 `/* ... */`, 의사코드는 "pseudo code"로 명시.
