# explore-topic

pancodev.io 블로그의 **다음 글 주제를 발굴하고 방향을 확정**하는 skill이다. 목표는 둘이다.

1. 사용자가 "쓸 게 없다"에서 "이걸 이렇게 쓰면 되겠다"로 넘어가게 만든다.
2. 그 결과를 `write-article` skill이 곧바로 이어받을 수 있는 형태(아티클 브리프)로 남긴다.

방식의 뿌리는 grilling이다 — 끊임없이, 한 번에 하나씩 묻는다. 다만 grilling과 결정적으로 다른 점은
**중립적으로 캐묻기만 하지 않고, 매 갈림길에서 블로그 맥락에 근거해 방향을 능동적으로 추천한다**는 것이다.
사용자는 추천을 받아들이거나, 반박하거나, 다른 길을 고른다. 결정은 언제나 사용자 몫이다.

이 skill은 명시적으로 호출될 때만 동작한다.

## 시작 전에 반드시 로드할 것

초안 주제를 던지기 전에 아래를 **먼저 확보한다**. 추천의 근거는 전부 여기서 나온다.

- `docs/article-index.md` (아티클 책갈피) — **현황을 한눈에 분석**하는 출발점. 전체/카테고리별 개수,
  연도별 발행 추이, 최근 발행 흐름이 이미 집계돼 있다. "어느 카테고리가 얇은가", "요즘 뜸했던 영역은
  어디인가", "최근 어떤 흐름으로 써왔는가"를 빠르게 파악해 추천 각도를 잡는 데 쓴다. 단, 이 문서는
  수동 갱신 스냅샷이라 낡아 있을 수 있으니 최종 사실 확인은 아래 `content/article/`로 한다.
- `content/article/` 하위를 훑어 **이미 쓴 글 목록과 카테고리 분포**를 파악한다. (무엇이 다뤄졌고,
  어느 카테고리가 비어 있는지 = 추천의 핵심 재료. 사실 조회이므로 사용자에게 묻지 말고 직접 읽는다.
  책갈피와 실제 파일이 어긋나면 파일이 정답이다.)
- `.kiro/skills/write-article/references/voice.md` — 이 블로그가 어떤 글을 "사람 냄새 나는 좋은 글"로
  보는지. 주제가 이 voice로 소화 가능한지 판단하는 기준.
- `.kiro/skills/write-article/references/structure.md` — 글 유형(고찰/튜토리얼/회고/오피니언/히스토리)과
  구조. 논지 전개를 확정할 때 이 유형 분류를 쓴다.
- `.kiro/skills/explore-topic/references/recommendation.md` — 주제를 발굴·평가·추천하는 휴리스틱.

## 대화 원칙 (grilling에서 계승 + 확장)

- **한 번에 하나씩 묻는다.** 여러 질문을 쏟아내면 사용자가 길을 잃는다. 답을 받고 나서 다음으로 간다.
- **사실은 직접 찾고, 결정만 묻는다.** "react 카테고리에 리렌더링 글이 몇 개인가" 같은 사실은
  파일을 읽어 확인한다. "어떤 각도로 쓸까" 같은 판단만 사용자에게 넘긴다.
- **매 질문에 추천안을 붙인다.** 그냥 "어떻게 생각해요?"가 아니라 "나는 A를 추천한다. 이유는 ~.
  다만 B도 가능한데 그 경우 ~. 어느 쪽인가?"로 묻는다. 근거는 블로그 맥락(빈 카테고리, voice 적합성,
  독자 가치, 사용자 경험의 밀도)에서 끌어온다.
- **얕으면 파고든다.** 답이 "그냥 정리 차원"처럼 재료가 없으면 한 번 더 캔다. 좋은 주제는 구체적
  경험·의문·마찰에서 나온다.
- **합의 전엔 write-article로 넘기지 않는다.** 사용자가 "이 주제, 이 방향으로 가자"라고 확정하기 전까지
  브리프를 완성하거나 글쓰기로 넘어가지 않는다.

## 워크플로우

### 1단계 — 재료 스캔 (사용자 안에서 캐기)

사용자에게 최근의 **경험·마찰·의문**을 꺼내게 한다. 주제는 지어내는 게 아니라 사용자 안에서 발굴한다.
`AskUserQuestion`으로 한두 개씩, 흐름을 타며 묻는다. 예시 물음:

- 최근 실무에서 삽질했거나, 처음 알게 됐거나, 판단이 갈렸던 순간이 있었나?
- 멘토링/동료와의 대화에서 반복해서 나온 질문이나 오해가 있었나?
- 요즘 관심이 가는 기술·논쟁·트렌드가 있나? 그게 "내 경험"과 닿아 있나?

동시에 백그라운드로 `content/article/`을 읽어 이미 다룬 주제·카테고리 분포를 파악해 둔다.

### 2단계 — 후보 주제 발굴 + 추천

1단계 재료와 기존 글 분석을 합쳐, **후보 주제 2~4개**를 제시한다. 각 후보마다:

- 한 줄 주제 + 어느 카테고리에 들어갈지
- 이 블로그에서 왜 쓸 만한가 (빈 영역인가 / 사용자 경험이 진한가 / 독자 가치가 있는가)
- 어떤 글 유형에 어울릴지 (고찰/튜토리얼/회고/오피니언/히스토리)
- 예상되는 약점 (재료 부족? 이미 비슷한 글이 있음? voice로 살리기 어려움?)

그리고 그중 **하나를 추천**하고 이유를 밝힌다. (평가 기준은 `recommendation.md`.)
사용자가 후보를 고르거나, 합치거나, 완전히 새 방향을 제시하면 반영한다.

### 3단계 — 그릴링으로 방향 좁히기

주제 후보가 하나로 좁혀지면, 이제 **논지 전개**를 벼린다. 한 번에 하나씩, 추천을 붙여 묻는다.
결정 트리를 순서대로 내려가며 의존성을 해소한다. 다뤄야 할 결정들:

- **동기(왜 이 글인가)** — 서문 첫 문단이 될 구체적 계기. 얕으면 더 캔다.
- **핵심 주장 / 남길 한 문장** — 독자가 덮을 때 가져갈 것. 단정할지 열어둘지도 함께 정한다.
- **글 유형과 골격** — structure.md의 유형 중 무엇인가. 그에 맞는 섹션 뼈대(`####` 목록)를 추천한다.
- **타깃 독자와 깊이** — 주니어냐 실무자냐에 따라 설명 수위가 갈린다.
- **차별점** — 기존 글/외부 글과 무엇이 다른가. (이미 있는 글과 겹치면 각도를 틀거나 후보를 바꾼다.)
- **참고 소스** — 인용할 공식문서/소스코드/통계가 있는가. (이 블로그는 인라인 링크 밀도가 높다.)

각 결정에서 "나라면 이렇게 가겠다"는 추천을 먼저 놓고 사용자 확답을 기다린다.

### 4단계 — 아티클 브리프 확정 (합의 지점)

위 결정들이 모이면 아래 형식으로 **브리프를 정리해 사용자에게 최종 확인**받는다.
이 브리프가 곧 `write-article` 1단계 인터뷰의 답이 된다.

```
## 아티클 브리프
- 제목안: (후킹되는 title 후보 1~2개)
- 카테고리: (content/article/<category>)
- slug 안: (영문 소문자 + 하이픈)
- 글 유형: (고찰 / 튜토리얼 / 회고 / 오피니언 / 히스토리)
- 동기(서문): (구체적 계기 한두 줄)
- 핵심 주장: (남길 한 문장 + 결론을 단정할지/열어둘지)
- 타깃 독자: (누구, 설명 깊이)
- 섹션 골격: (#### 소제목 목록 + 각 줄 한 줄 요약)
- 차별점: (기존/외부 글과의 차이)
- 참고 소스: (링크·이미지 목록)
- 미확보 재료: (write-article 인터뷰에서 더 캐야 할 것)
```

사용자가 "이대로 가자"라고 확정하기 전에는 다음 단계로 넘어가지 않는다.

### 5단계 — write-article로 핸드오프

브리프가 확정되면 `write-article` skill을 이어서 실행한다.

- 확정된 브리프를 write-article의 1단계 인터뷰 입력으로 전달한다. 이미 채워진 항목은 다시 묻지 않고,
  브리프의 "미확보 재료"만 write-article 인터뷰에서 마저 캔다.
- 이후는 write-article의 워크플로우(아웃라인 승인 → 초안 → voice 셀프리뷰 → 파일 생성)를 따른다.
- 사용자가 "지금은 주제만 정하고 글은 나중에"라고 하면, 브리프만 남기고 여기서 멈춘다.

## 하지 말 것

- 재료를 캐지 않고 일반론적 주제("리액트 최적화 총정리")를 던지지 않는다. 주제는 사용자 경험에서 나온다.
- 여러 질문을 한꺼번에 쏟아붓지 않는다. (grilling의 핵심 규율이다.)
- 추천 없이 "뭐 쓰고 싶어요?"만 반복하지 않는다. 이 skill의 존재 이유가 능동적 방향 제시다.
- 사용자 확정 없이 브리프를 완성하거나 write-article로 넘어가지 않는다.
- 이미 블로그에 있는 글과 겹치는 주제를, 차별점 없이 추천하지 않는다. (2단계에서 기존 글을 반드시 확인.)
- 사실로 확인 가능한 것(기존 글 목록, 카테고리)을 사용자에게 묻지 않는다. 직접 읽는다.
