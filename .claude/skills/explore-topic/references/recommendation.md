# recommendation.md — 주제 발굴·평가·추천 휴리스틱

이 문서는 `explore-topic`이 후보 주제를 **찾고, 점수 매기고, 추천**할 때 쓰는 기준이다.
추천은 감이 아니라 아래 근거에 기반해야 한다. "왜 이 주제를 추천하는가"를 항상 이 기준으로 설명한다.

---

## 좋은 주제의 4가지 신호

이 블로그(pancodev.io)에서 잘 굴러가는 주제는 대체로 아래를 만족한다. 후보를 볼 때 이 4개로 검산한다.

1. **경험 밀도** — 사용자가 직접 겪은 마찰·삽질·판단·의문이 있는가.
   voice.md가 요구하는 "사람 냄새"의 원천. 이게 없으면 아무리 좋은 주제도 AI향 글이 된다. **가장 중요.**
2. **독자 가치** — 읽는 사람이 가져갈 게 있는가. 실무 문제 해결, 오해 교정, 새로운 관점 중 하나.
3. **차별점** — 이미 널린 글/기존 블로그 글과 각도가 다른가. "총정리"류가 아니라 "나만의 진입각"이 있는가.
4. **voice 적합성** — 반말체·1인칭·열린 결론·취소선 자조로 소화 가능한 소재인가.
   지나치게 건조한 레퍼런스성 주제는 이 블로그 voice와 잘 안 붙는다.

## 기존 글과의 관계로 후보 분류

`content/article/`을 읽어 이미 있는 글을 파악한 뒤, 후보를 아래 셋 중 하나로 분류해 추천 강도를 정한다.

- **빈 영역 (강력 추천)** — 카테고리는 있는데 안 다룬 주제, 또는 다룬 적 없는 새 카테고리.
- **심화·후속 (추천)** — 기존 글의 다음 편, 더 깊은 파기, 반대 관점. (예: "1편"이 있으면 "2편")
  기존 글을 인라인 링크로 엮을 수 있어 블로그 응집도가 올라간다.
- **중복 위험 (비추천 또는 각도 전환 필요)** — 기존 글과 주제·결론이 겹침.
  이 경우 각도를 틀거나(오피니언 → 튜토리얼 등), 다른 후보를 권한다.

현재 카테고리: `develop` / `next` / `react` / `remix` / `web` (신규 카테고리도 가능).
분포를 보고 "react에 글이 몰려 있고 develop이 비어 있다" 같은 관찰을 추천 근거로 쓴다.

## 후보 → 글 유형 매핑 가이드

재료의 성격으로 어울리는 유형을 제안한다. (유형별 구조는 structure.md.)

| 사용자가 가진 재료                                                  | 어울리는 유형 |
| ------------------------------------------------------------------- | ------------- |
| "왜 이렇게 동작하지?" 하는 풀리지 않은 의문, 소스코드를 파고든 경험 | 고찰형        |
| 실무에서 특정 문제를 해결한 단계적 과정, 마주친 에러·버전·설정      | 튜토리얼형    |
| 특정 시기·프로젝트·성장에 대한 돌아봄, 감정과 배움                  | 회고형        |
| 통념에 대한 반론, "이건 오해다"라는 주장, 벤치마크·통계             | 오피니언형    |
| 기술의 등장 배경·흐름·뒷이야기에 대한 관심                          | 히스토리형    |

애매하면 사용자에게 "이 소재는 A(고찰)로도, B(튜토리얼)로도 갈 수 있다. 나는 A를 추천하는데 ~때문이다. 어느 쪽?"으로 되묻는다.

## 추천 문장 템플릿

매 갈림길에서 이 골격으로 추천한다. 근거 없는 추천은 하지 않는다.

> "나는 **[선택지]** 를 추천한다. 이유는 [블로그 맥락 근거 — 빈 영역/경험 밀도/독자 가치/차별점].
> 다만 [대안]도 가능한데, 그 경우 [트레이드오프]. 어느 쪽으로 갈까?"

## 얕은 답을 깊게 파는 후속 질문

재료가 부족한 신호와, 그때 던질 후속 질문:

- "그냥 정리 차원" → "정리하고 싶어진 계기가 뭐였나? 최근에 이걸 다시 들여다본 이유가 있었을 텐데."
- "요즘 X가 유행이라" → "X를 직접 써봤나? 써보며 기대와 달랐던 지점이 있었나?"
- "이게 중요한 것 같아서" → "누구에게, 왜 중요한가? 그렇게 느낀 본인의 경험이 있나?"

경험으로 착지하지 못하는 주제는, 아무리 트렌디해도 추천을 보류하고 다른 후보로 유도한다.

## 추천할 때 경계할 것

- 사용자가 안 겪은 걸 "이런 것도 있어요" 식으로 밀어붙이지 않는다. (지어낸 경험은 voice에서 즉시 티가 난다.)
- "총정리", "완벽 가이드", "한 방에 끝내는" 류의 포괄 주제로 유도하지 않는다. 좁고 구체적일수록 좋다.
- 한 번에 후보를 5개 이상 나열해 사용자를 압도하지 않는다. 2~4개로 압축한다.
