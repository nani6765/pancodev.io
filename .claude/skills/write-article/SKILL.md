# write-article

pancodev.io 블로그의 새 기술 아티클을 작성하는 skill이다. 이 skill의 존재 이유는 단 하나 —
**판다코딩(panco)이 직접 쓴 것처럼 읽히는 글**을 만드는 것이다. 정보 정확성보다 앞서는 목표가
"AI가 쓴 티가 안 나는가"이다. 그래서 voice 가이드를 최우선 기준으로 삼는다.

이 skill은 명시적으로 호출될 때만 동작한다. 자동으로 끼어들지 않는다.

## 시작 전에 반드시 로드할 것

작업을 시작하면 아래 세 파일을 **먼저 읽는다**. 특히 `voice.md`는 초안을 한 글자라도 쓰기 전에
반드시 숙지한다.

- `references/voice.md` — 문체·톤·시그니처. AI향 제거 체크리스트. **가장 중요.**
- `references/structure.md` — 글 유형별 구조 템플릿, front-matter 규칙, 콜아웃·이미지 관례
- `references/examples.md` — 기존 16개 아티클에서 뽑은 ✓/✗ 실제 문장 대조

## 워크플로우

전체 5단계다. 각 단계를 건너뛰지 말고, 특히 2단계 아웃라인 승인과 4단계 셀프리뷰는 생략하지 않는다.

### 1단계 — 인터뷰 (AskUserQuestion으로 drill)

좋은 글은 좋은 재료에서 나온다. 초안을 바로 뽑지 말고, `AskUserQuestion`으로 아래를 캐낸다.
한 번에 다 묻지 말고 흐름에 맞춰 2~3개씩 묶어 묻되, **답이 얕으면 한 번 더 파고든다**.
(개인 경험·맥락이 얕으면 글에서 AI향이 난다. 이 인터뷰가 그걸 막는 유일한 방어선이다.)

캐내야 할 것:

1. **주제와 카테고리** — 무엇에 대한 글인가. `content/article/` 아래 어느 카테고리인가
   (`develop`/`next`/`react`/`remix`/`web`, 또는 새 카테고리). 카테고리 폴더명이 front-matter
   `category`와 slug 경로를 결정한다.
2. **글 유형** — 고찰형 / 튜토리얼형 / 회고형 / 오피니언형 / 히스토리형 중 무엇에 가까운가.
   (유형별 구조는 `structure.md` 참고.) 유형이 정해져야 서문·결론의 톤이 정해진다.
3. **동기 (왜 이 글을?)** — 이게 서문의 첫 문단이 된다. "회사에서 ~할 일이 생겼다",
   "멘토링하다 가장 많이 받는 질문이라", "동료가 ~라고 말해줬는데 충격이었다" 처럼
   **구체적인 계기**를 반드시 확보한다. "정리 차원에서"는 재료가 아니다. 더 파고든다.
4. **실무 맥락과 경험** — 직접 겪은 시행착오, 삽질, 판단, 회사 상황(B2B/B2C 등),
   실제로 마주친 에러 메시지·버전·수치. 글의 신뢰도와 "사람 냄새"는 여기서 나온다.
5. **핵심 주장 / 남기고 싶은 결론** — 독자가 이 글을 덮을 때 가져갈 한 문장.
   단, 결론을 단정할지 열어둘지는 유형에 따라 다르다(`voice.md`의 결론 파트 참고).
6. **타깃 독자** — 주니어/실무자/특정 스택 사용자 등. 설명 깊이를 조절한다.
7. **참고 링크·이미지** — 인용할 공식문서/MDN/GitHub 소스/외부 아티클, 넣을 이미지가 있는지.
   (이 블로그는 인라인 링크가 매우 풍부하다. 링크할 소스를 미리 확보한다.)

### 2단계 — 아웃라인 제시 후 승인

인터뷰 재료로 아웃라인을 만들어 사용자에게 제시하고 **승인을 받는다**. 승인 없이 본문으로 넘어가지 않는다.

아웃라인에 담을 것:
- 제목안 (title) — 후킹되는지 확인
- 서문에서 풀 동기 한두 줄
- `####` 소제목 목록과 각 섹션이 다룰 내용 한 줄 요약
- 결론의 방향 (단정 / 열린 결말 / 한계 인정)
- 삽입할 코드블록·이미지·인용 링크의 위치

사용자가 섹션 순서·범위·깊이를 조정하면 반영하고 다시 확인한다.

### 3단계 — 섹션별 초안 작성

승인된 아웃라인을 따라 본문을 쓴다. 쓰는 내내 `voice.md`를 옆에 두고 문장마다 대조한다.
구조·front-matter·콜아웃·이미지 경로는 `structure.md` 규칙을 그대로 따른다.

핵심 원칙(자세한 건 `voice.md`):
- 평서형 반말체("~다", "~해보자", "~하자"), 1인칭("나")
- `~~취소선~~`으로 자조·솔직한 속내를 시그니처처럼 섞되 과용하지 않는다
- 독자에게 질문을 던진다, 공식문서·소스코드를 인라인 링크로 촘촘히 건다
- 실제 코드는 실무 코드로, 주석은 한국어로, 생략은 `/* ... */`

### 4단계 — voice 셀프리뷰 (AI향 제거)

초안을 다 쓰면 **반드시** `voice.md`의 "AI향 제거 체크리스트"로 전문을 훑는다.
걸리는 문장은 그 자리에서 고친다. 대표적으로 잡아낼 것:
- "결론적으로", "총정리하면", "~를 살펴보겠습니다"의 영혼 없는 나열
- 모든 문단이 비슷한 길이·구조로 반복되는 리듬
- 불릿·굵게의 남발, 근거 없는 일반론으로 마무리하는 결론
- 개인 경험·구체 수치·실제 링크가 빠진 붕 뜬 설명

`examples.md`의 ✗ 사례와 대조해 "이건 초기 글의 AI향 결론과 똑같지 않은가"를 자문한다.

### 5단계 — 파일 생성 및 검증

`content/article/<category>/<slug>.md` 파일을 실제로 생성한다.

- **slug** = 파일명(확장자 제외) = URL 경로. 영문 소문자 + 하이픈.
- **front-matter** 5개 필드를 채운다: `title`, `description`, `keywords`, `category`, `created_at`.
  - `category`는 폴더명과 **반드시 일치**.
  - `created_at`은 `YYYY-MM-DD HH:mm`. 예약 발행하려면 미래 시각(KST)으로. (기본은 현재 시각.)
  - `title`에 콜론(`:`)·특수문자가 있으면 큰따옴표로 감싼다.
  - `description`은 SEO용 요약(영문/국문은 기존 글 관례를 따르되 사용자에게 확인).
- 이미지가 있으면 `content/article/<category>/image/<slug>/`에 두고 상대경로
  `image/<slug>/파일명`으로 참조하도록 안내한다.
- 마지막으로 `structure.md`의 체크리스트로 검증하고, 사용자에게 `yarn dev`로 확인하라고 안내한다.

## 하지 말 것

- 인터뷰·아웃라인 승인 단계를 건너뛰고 바로 완성 초안을 던지지 않는다.
- 존댓말체로 본문을 쓰지 않는다. (사용자와의 대화는 존댓말, **글 본문은 반말체**다.)
- `tech-blog-article` skill의 Toss/Kakao 스타일을 끌어오지 않는다. 이 블로그의 voice는 독립적이다.
- 겪지 않은 경험을 지어내지 않는다. 재료가 부족하면 1단계로 돌아가 더 캐낸다.
